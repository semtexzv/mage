# SDK Crate Structure

The `mage` crate (`pkg/sdk`) is the single dependency for extension authors
and the top-level namespace for the host binary. It re-exports the full
module tree — no feature gates, no optional deps. Every build has the same
capabilities.


## Module Tree

```rust
// pkg/sdk/src/lib.rs

/// Extension trait, hooks, event/result types, factory registry.
pub use mage_core::extension;

/// Agent message types, events, delivery modes.
pub use mage_core::types;

/// Session handle and spawn.
pub use mage_core::session;

/// Agent loop (usually not needed by extensions directly).
pub use mage_core::agent_loop;

/// Application layer: commands, input routing, session lifecycle.
pub use mage_app as app;

/// Agent event stream types.
pub use mage_core::event_stream;

/// LLM abstraction: Provider trait, Message, Model, CancelToken, events.
pub use llm;

/// Ref-counted strings: Str, Str<Atomic>, Local, Atomic, Mode.
pub use refstr;

/// Terminal UI: renderer, markdown, editor, keymap, styles.
pub use mage_tui as tui;

/// Dynamic workspace compiler.
pub use mage_build as build;
```


## Prelude

`use mage::prelude::*` gives extension authors everything needed to write
a tool or extension hook.

```rust
pub mod prelude {
    // Extension system
    pub use crate::extension::{
        Extension, ExtensionFactory, ExtensionRegistry, ExtensionContext,
        LoopHandle, LoopCommand, ToolHandle,
        AgentEndEvent, TurnStartEvent, TurnEndEvent, ContextEvent,
        ToolCallEvent, ToolResultEvent, InputEvent,
        ContextResult, ToolCallResult, ToolResultResult, InputResult,
        BeforeAgentStartEvent, BeforeAgentStartResult,
    };
    pub use async_trait::async_trait;

    // Session
    pub use crate::session::SessionHandle;
    pub use crate::session::spawn as spawn_session;

    // Agent loop
    pub use crate::agent_loop::{AgentLoop, LoopError};

    // Agent types
    pub use crate::types::{Message, MessageBody, EntryId, AgentEvent, DeliverAs, ToolResult, ToolUpdate};

    // LLM types (the subset extensions actually touch)
    pub use crate::llm::CancelToken;
    pub use crate::llm::types::{
        UserContent, ContentBlock, Model, Usage,
    };

    // Auth
    pub use crate::llm::{AuthStatus, Authenticator, LoginStep, LoginReceiver};

    // Strings
    pub use refstr::Str;
    pub use crate::app::command::{Command, CommandRegistry, CommandError};
    pub use crate::app::app::{App, AppError};
}
```


## Cargo.toml
During development the workspace uses path deps:
```toml
[package]
name = "mage"
version = "0.1.0"
edition = "2024"
rust-version = "1.85"

[dependencies]
mage-core = { path = "../core" }
mage-llm = { path = "../llm" }
mage-tui = { path = "../tui" }
mage-build = { path = "../build" }
refstr = { path = "../refstr" }
```

The published crate on crates.io uses version deps:
```toml
[package]
name = "mage"
version = "0.1.0"
edition = "2024"
rust-version = "1.85"

[dependencies]
mage-core = "0.1"
mage-llm = "0.1"
mage-tui = "0.1"
mage-build = "0.1"
refstr = "0.1"
```


## SDK Distribution

The `mage` crate and all sub-crates (`mage-core`, `mage-llm`, `mage-tui`,
`mage-build`, `refstr`) are published to crates.io. The full surface is
published — including `tui` and `build` — because the generated binary IS
the SDK consumer. There is no separate `main.rs` outside the SDK; the
template generates everything, and the generated code depends on the full
stack.

The mage binary embeds two constants identifying which SDK it was built
against:
```rust
pub const MAGE_SDK_VERSION: &str = "0.1.0";
pub const MAGE_SDK_HASH: &str = "sha256:a1b2c3d4...";
```

The SDK dependency is **implicit** for extensions. Extension authors do not
write `@dep mage = ...` — mage-build injects it automatically at the
version the binary was built against. Extensions that need to pin a
different SDK version (rare) can write `@dep mage = "X.Y.Z"` to override.

The generated staging `Cargo.toml` uses a registry dep:
```toml
[dependencies]
mage = "0.1.0"
```
No core crate sources in staging. No symlinks. Cargo fetches the SDK from
its registry cache.

## Generated Binary
The generated binary is **not** a static `src/main.rs`. It is
code-generated by mage-build at compile time: the `main()` function,
extension registration, and provider wiring are all synthesized from the
current set of discovered extensions and the Jinja2 template.

The generated code depends on `mage` (the SDK) as a crates.io registry
dependency, plus whatever providers and extensions are in scope. The exact
Cargo.toml is written into the staging directory by mage-build — it is not
a file in the repo.


## Usage Patterns

Extension author (single-file module). The SDK dependency is implicit —
mage-build injects it automatically. Authors just use the prelude:
```rust
use mage::prelude::*;

struct MyToolExt;

#[async_trait(?Send)]
impl Extension for MyToolExt {
    fn init(&mut self, reg: &mut ExtensionRegistry) {
        reg.tool(
            llm::Tool {
                name: "my_tool".into(),
                description: "Does a thing".into(),
                parameters: serde_json::json!({"type": "object"}),
            },
            |_id, params, _handle| async move {
                ToolResult::success("done")
            },
        );
    }
}
```

Extension author (explicit imports):
```rust
use mage::extension::{Extension, ExtensionRegistry, ExtensionContext, ToolCallEvent, ToolCallResult};
use mage::types::ToolResult;
use mage::llm::CancelToken;
use refstr::Str;
```

Host binary:
```rust
use mage::prelude::*;
use mage::tui::{App, run};
use mage::build::Bundle;
use mage::llm::Provider;
```


## Third-party Dependencies
The SDK does NOT re-export `serde_json`, `serde`, `tokio`, or `reqwest`.
All third-party crates are added by extensions via `@dep` annotations:
```rust
// @dep serde_json = "1"
// @dep serde = { version = "1", features = ["derive"] }
// @dep tokio = { version = "1", features = ["process", "fs"] }
// @dep reqwest = { version = "0.12", features = ["json"] }
```
These are resolved by mage-build during compilation. The versions are
included in the bundle hash via Cargo.lock (see DESIGN-REPRODUCIBLE-BUILDS.md).
`serde_json` is so universally needed that the built-in template includes
it as a dependency of every generated binary. Extension authors still add
`@dep serde_json = "1"` if they use it directly — mage-build deduplicates.


## SDK Version Management

The `mage sdk` subcommand manages the SDK version:

- `mage sdk current` — display the SDK version and build hash the binary
  was compiled against.
- `mage sdk upgrade` — check for the latest compatible SDK on crates.io,
  update `.mage/config.toml`, and rebuild the current extensions against
  the new version.
- `mage sdk upgrade 0.4.0` — upgrade to a specific version.
- `mage sdk pin 0.3.2` — lock the SDK version in `.mage/config.toml`.
  Subsequent builds will use this version regardless of what the binary
  was built against.

Project-level override in `.mage/config.toml`:
```toml
[sdk]
version = "0.3.2"   # pinned version; overrides binary default
```

When the binary detects a newer SDK is available (e.g. on startup or
periodically), it displays a soft notification: "SDK 0.4.1 available, run
`mage sdk upgrade` to update." This is informational, not a blocker.


## Snapshot Format

The snapshot no longer contains core crate sources. Contents:

- Extension module sources (`.rs` files from `.mage/ext/`).
- Generated `main.rs` (output of the Jinja2 template).
- `Cargo.lock` — pins the SDK version and all transitive dependencies.
  Rebuild from snapshot is deterministic via Cargo.lock resolution.
- `mage-snapshot.json` — metadata including `sdk_version`, `sdk_hash`,
  `parent_hash`, and per-module metadata (name, hash, dep annotations).

Rebuilding from a snapshot: extract, `cargo build`. Cargo resolves the SDK
from its registry cache using the pinned version in Cargo.lock. No source
distribution required.

## Rationale

**No feature gates.** Every mage binary includes the full stack: core,
LLM, TUI, build system. The published SDK exposes the full surface because
the generated binary IS the SDK consumer — it needs the TUI, the build
system, and everything else. Extensions compiled into the binary inherit
access to all of it. This simplifies the dependency graph and avoids
conditional compilation bugs. The cost is compile time for extensions that
don't use TUI or build — acceptable because mage-build caches aggressively
and these crates only compile once.
**`pub use mage_llm as llm`, not `pub mod llm`.** Straight re-export.
`mage::llm::Provider`, `mage::llm::types::Model`. No wrapper, no
indirection. Same for `tui` and `build`.
**Single dependency for extensions.** Extension authors get the SDK
automatically — mage-build injects `mage = "X.Y.Z"` as a crates.io
dependency at the version the binary was built against. Authors don't need
to know mage-core vs mage-llm vs refstr. The internal crate boundaries are
an implementation detail.

**No source distribution.** The binary carries a version string and build
hash, not SDK source code. Snapshots shrink dramatically — they contain
only extension sources, generated template output, and Cargo.lock. Rebuilds
work via Cargo.lock pinning against the crates.io registry.

**Implicit SDK dependency.** Extensions don't declare the SDK dep;
mage-build injects it. This eliminates boilerplate (`@dep mage = ...` in
every extension file) and prevents version skew — all extensions in a
build use the same SDK version unless explicitly overridden.

**Prelude is opinionated.** It exports everything an extension author
typically needs. Extension authors who want precision use explicit imports
from submodules. Both work.
